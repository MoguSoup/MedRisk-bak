# 模型说明

当前版本提供五类疾病的教学演示预测接口：糖尿病、心脏病、慢性肾病、肝病、中风。

模型服务接口采用统一 adapter 输出结构。默认稳定模型为 Logistic Regression、RandomForest、ExtraTrees、HistGradientBoosting 和 XGBoost；可选高级模型为 LightGBM、CatBoost、TabPFN、TabICL 和 FT-Transformer 能力入口：

- `risk_label`：低/中/高风险标签。
- `risk_probability`：风险概率。
- `confidence`：演示置信度。
- `top_factors`：Top 5 风险因素，模拟 SHAP 影响方向和贡献度。
- `recommendations`：健康建议。
- `model_version`：模型版本号。

当前实时推理优先使用管理员已启用的训练模型；若某病种没有启用训练产物，则回退到确定性教学基线，便于演示和测试。

默认基线现在按“已部署公开数据评估基线”展示，而不是空白占位模型。糖尿病和中风使用 CDC BRFSS 2024 派生训练/评估数据元信息；肝病使用 CDC NHANES 2017-March 2020 派生实验室/问卷数据元信息；心脏病与慢性肾病可使用 UCI Heart Disease 和 UCI Chronic Kidney Disease 公开数据作为补充评估来源。模型版本表和 `/models/public-evaluations` 会展示评估数据集、来源、样本量、公开链接和验证方式。仓库不提交大体量 `data/processed/*.csv`，需要复现训练时运行 `scripts/prepare-large-datasets.ps1` 重新生成训练/评估拆分和 `manifest.json`，再在管理后台“数据集管理”点击“导入公开训练数据集”注册，随后选择训练集和同病种评估集完成训练、评估、启用。

训练模型启用后，FastAPI 模型服务会把活动模型映射写入 `MEDRISK_ACTIVE_MODELS_FILE`；Docker Compose 默认值是 `/shared/models/training/active-models.json`，位于持久化 `models/` 挂载目录下。服务重启时会自动加载该文件中仍可读取的 `model.joblib`，从而真正部署到预测接口。

`GET /models/capabilities` 会返回各模型类型是否可用。默认依赖保证 `logistic_regression`、`random_forest`、`extra_trees`、`hist_gradient_boosting` 和 `xgboost` 可用；如需试用 `lightgbm`、`catboost`、`tabpfn`、`tabicl` 或后续 FT-Transformer 运行时，可在 conda `MedRisk` 环境中按需安装 `medrisk_model_service/requirements-advanced.txt`。高级依赖缺失时能力接口会返回 unavailable reason，不影响默认模型训练、预测和项目启动。

## 智能问答与 LLM

智能问答使用 Spring Boot 后端内置的 GraphRAG 流程，不依赖 FastAPI 风险预测模型服务。后端先从 Neo4j 知识图谱、疾病信息和病历案例中检索上下文，再调用 OpenAI-compatible Chat Completions 接口生成回答。本地 Windows 演示必须自行提供本机 Neo4j，默认检查 `bolt://localhost:7687`；不会自动连接 `med_tencent`，也不应为了本地演示开放远端 Neo4j Bolt。Linux `med_tencent` 部署由 Docker Compose 内部网络连接 `bolt://neo4j:7687`。

本地 `.env` 不应保留 Compose 内部地址 `bolt://neo4j:7687`；该地址只适用于远端 Docker Compose。智能问答若遇到图谱不可达，会继续使用疾病档案、病历案例和本地兜底回答，但图谱管理页仍会明确显示 Neo4j 未连接。

默认配置项：

- `MEDRISK_LLM_BASE_URL`：默认 `https://dashscope.aliyuncs.com/compatible-mode/v1`
- `MEDRISK_LLM_API_KEY`：Qwen/百炼或兼容服务 API Key，留空时不调用外部模型
- `MEDRISK_LLM_MODEL`：默认 `qwen-plus`
- `MEDRISK_VISION_MODEL`：视觉/全模态模型，默认 `qwen3.5-omni-plus-2026-03-15`，图片输入时会自动使用
- `MEDRISK_IMAGE_MODEL`：图片生成模型，默认 `wanx-v1`，仅在用户打开“输出图片”时调用

阿里云百炼 workspace 专属域名同样使用 OpenAI-compatible 格式，例如 `https://your-workspace.cn-beijing.maas.aliyuncs.com/compatible-mode/v1`。真实 API Key 只写入 gitignored `.env` 或服务器 `.env`，不要写入源码、README、文档、`.env.example` 或前端环境变量。

同一个百炼 API Key 可同时用于文本、视觉/全模态和图片生成。系统会按场景自动切换：纯文本默认使用 `MEDRISK_LLM_MODEL`，图片输入使用 `MEDRISK_VISION_MODEL`，打开“输出图片”时调用 `MEDRISK_IMAGE_MODEL`。当 `MEDRISK_LLM_API_KEY` 为空或外部接口不可用时，系统会基于检索到的图谱、疾病和病历内容生成可演示的规则兜底答案，并保留“仅用于教学演示，不能替代医生诊断”的提示；图片生成失败时会在回答中提示降级，不影响文字回答保存。

问答检索会遵守后端可见性策略：患者只检索公开知识，医生可检索公开、医生专用和自己提交的草稿，管理员可检索全部知识。回答区会展示疾病、病历、图谱上下文、来源标识、`usedModel`、`provider`、`fallbackUsed` 和 `evidenceSources`，医生专用引用不会泄露给患者。

外部模型调用前会先进行平台范围判断。只有医疗健康、风险预测、医学知识库、病历/文档、知识图谱、模型训练评估和平台使用相关问题会进入检索与 LLM；明显无关问题会直接返回范围提示，`usedModel=policy-guard`、`provider=medrisk-policy`，不消耗外部模型调用。

知识图谱构建不在运行时调用外部搜索 API。后端使用离线确定性规则从标题、摘要和正文中抽取疾病、症状、风险因素、检查、治疗、药物、文本单元、关键词和观点节点，并为每篇文档写入文档级唯一 `graphKey`，避免不同文档同名实体互相覆盖。单篇非空文档构建结果至少包含 50 个文档相关节点，便于 GraphRAG 检索和可视化首屏展示。

## 演示知识与数据集

中等演示包用于让 GraphRAG 和训练管理在空库中也能演示。后端会幂等补齐约 50 个疾病、100 篇知识文档、50 个合成病历和 5 组以上训练/评估 CSV。公开来源只保存可复用摘要、链接和来源元数据；授权不明确的网页正文不会批量复制进库。

主要来源边界：

- MedlinePlus：健康主题摘要、MeSH/相关主题和来源链接。
- Disease Ontology / HPO：疾病分类、表型/症状和本体链接。
- UCI Heart Disease、Chronic Kidney Disease 等：演示训练/评估数据集，统一转换为含 `label` 列的 CSV。
- NHANES：公共健康、营养、实验室和风险因素字段的派生演示数据。
- openFDA：药品标签链接型知识。
- Synthea 风格合成病历：仅用于模拟 EHR 场景，全部标记为合成数据，非真实患者。

## 本地模型服务环境

本地 PowerShell 启动脚本固定使用 conda `MedRisk` 环境运行模型服务。脚本默认兼容 `D:\IDEA\Anaconda\envs\MedRisk\python.exe`，也会在其他电脑上通过 `MEDRISK_CONDA_ROOT`、当前激活的 `MedRisk` 环境、常见 Anaconda/Miniconda 目录和 `conda env list` 发现同名环境。脚本不创建、不激活 `medrisk_model_service\.venv`，也不从随机 `PATH` Python 启动，并通过 `python -s` 与 `PYTHONNOUSERSITE=1` 降低用户级 Python 包对模型服务的干扰。

模型服务默认监听 `127.0.0.1:8090`，健康检查为 `GET /health`。依赖安装输出写入 `logs/model-service-install.log`，服务 stdout/stderr 和启动进程记录均由脚本写入项目 `logs/` 目录；本地启动脚本会等待该健康检查、后端健康检查和前端页面都可访问后才输出启动完成。

邮箱验证码和忘记密码邮件由 Spring Boot 后端通过 SMTP 发送，不依赖 FastAPI 模型服务。相关环境变量为 `MEDRISK_MAIL_HOST`、`MEDRISK_MAIL_PORT`、`MEDRISK_MAIL_USERNAME`、`MEDRISK_MAIL_PASSWORD`、`MEDRISK_MAIL_FROM`、`MEDRISK_MAIL_SSL`、`MEDRISK_MAIL_STARTTLS`、`MEDRISK_MAIL_AUTH`；163 邮箱使用授权码而不是网页登录密码，推荐 `465 + SSL`。未配置时模型服务不受影响，发送验证码接口会返回明确业务错误。

本地后端 `demo` profile 使用 H2 文件库，并仅在该本地演示路径关闭 Flyway checksum 校验，以免旧演示库迁移记录阻塞登录相关接口启动；Docker Compose/MySQL 部署仍使用默认迁移校验。

启动故障防复发：前端登录 500 通常不是模型服务问题，而是后端 8080 未启动。若 `logs/backend.out.log` 出现 Flyway checksum validation failed，说明本地 H2 演示库保留了旧迁移记录；必须保留 `application-demo.yml` 中的本地 H2 Flyway 宽松校验，并继续通过 conda `MedRisk` 和 `mvn clean spring-boot:run -Dspring-boot.run.profiles=demo` 启动。

## 管理员训练流程

1. 管理员上传结构化 CSV 或 ZIP 数据集，数据集必须包含 `label` 二分类目标列。
2. Spring Boot 将数据集保存到 `uploads/datasets/` 并记录校验结果；也可以从 `MEDRISK_PUBLIC_DATASET_DIR` 导入脚本生成的公开训练/评估 CSV。
3. 管理员创建训练任务后，FastAPI 读取共享数据集并按 `modelType` 选择 adapter：`logistic_regression`、`random_forest`、`extra_trees`、`hist_gradient_boosting`、`xgboost`，以及可选 `lightgbm`、`catboost`、`tabpfn`、`tabicl`。请求会保存实际使用的 `hyperparameters`，并可携带同病种 `evaluationDatasetId` 自动执行公开评估。
4. 训练产物写入 `models/training/job-{id}/`，包含 `model.joblib`、`history.json`、`metadata.json`。
5. 训练成功后后端写入新的 `model_versions` 记录，管理员可在管理后台启用该版本。
6. 启用后模型服务加载 `model.joblib`，对应病种预测优先走训练模型。

XGBoost 训练曲线来自 `evals_result()`，前端展示 train/valid logloss 与 error；非迭代模型不伪造 epoch 曲线，改为展示 Accuracy、Precision、Recall/Sensitivity、Specificity、F1、ROC-AUC、PR-AUC、LogLoss、Brier Score、混淆矩阵、ROC/PR/校准曲线和样本预测摘要。不同模型不共用一套超参数：线性模型使用 `cValue/maxIterations/classWeight`，树集成模型使用 `nEstimators/maxDepth/classWeight/seed`，Boosting 模型使用 `nEstimators/maxDepth/learningRate/subsample/colsampleBytree/regLambda`，TabPFN 使用 `maxTrainSamples/device/ensembleSize`，TabICL 使用 `contextSize/maxTrainSamples/device/seed`。
