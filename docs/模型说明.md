# 模型说明

当前版本提供五类疾病的教学演示预测接口：糖尿病、心脏病、慢性肾病、肝病、中风。

模型服务接口采用与 XGBoost/LightGBM/CatBoost/RandomForest 兼容的输出结构：

- `risk_label`：低/中/高风险标签。
- `risk_probability`：风险概率。
- `confidence`：演示置信度。
- `top_factors`：Top 5 风险因素，模拟 SHAP 影响方向和贡献度。
- `recommendations`：健康建议。
- `model_version`：模型版本号。

当前实时推理优先使用管理员已启用的训练模型；若某病种没有启用训练产物，则回退到确定性教学基线，便于演示和测试。

## 智能问答与 LLM

智能问答使用 Spring Boot 后端内置的 GraphRAG 流程，不依赖 FastAPI 风险预测模型服务。后端先从 Neo4j 知识图谱、疾病信息和病历案例中检索上下文，再调用 OpenAI-compatible Chat Completions 接口生成回答。

默认配置项：

- `MEDRISK_LLM_BASE_URL`：默认 `https://dashscope.aliyuncs.com/compatible-mode/v1`
- `MEDRISK_LLM_API_KEY`：Qwen/百炼或兼容服务 API Key，留空时不调用外部模型
- `MEDRISK_LLM_MODEL`：默认 `qwen-plus`
- `MEDRISK_VISION_MODEL`：可选 Qwen-VL/兼容视觉模型

当 `MEDRISK_LLM_API_KEY` 为空或外部接口不可用时，系统会基于检索到的图谱、疾病和病历内容生成可演示的规则兜底答案，并保留“仅用于教学演示，不能替代医生诊断”的提示。图片问答始终支持上传和展示；仅在同时配置 API Key 与 `MEDRISK_VISION_MODEL` 时才会把图片传给视觉模型，否则回答会明确说明图片已保存但未做视觉识别。

问答检索会遵守后端可见性策略：患者只检索公开知识，医生可检索公开、医生专用和自己提交的草稿，管理员可检索全部知识。回答区会展示疾病、病历、图谱上下文和来源标识，医生专用引用不会泄露给患者。

## 演示知识与数据集

中等演示包用于让 GraphRAG 和训练管理在空库中也能演示。后端会幂等补齐约 50 个疾病、100 篇知识文档、50 个合成病历和 5 组以上训练/评估 CSV。公开来源只保存可复用摘要、链接和来源元数据；授权不明确的网页正文不会批量复制进库。

主要来源边界：

- MedlinePlus：健康主题摘要、MeSH/相关主题和来源链接。
- Disease Ontology / HPO：疾病分类、表型/症状和本体链接。
- UCI Heart Disease、Chronic Kidney Disease 等：演示训练/评估数据集，统一转换为含 `label` 列的 CSV。
- NHANES：公共健康、营养、实验室和风险因素字段的派生演示数据。
- openFDA：药品标签链接型知识。
- Synthea 风格合成病历：仅用于模拟 EHR 场景，全部标记为合成数据，非真实患者。

## 本地模型服务环境

本地 PowerShell 启动脚本固定使用 conda `MedRisk` 环境运行模型服务。脚本默认兼容 `D:\IDEA\Anaconda\envs\MedRisk\python.exe`，也会在其他电脑上通过 `MEDRISK_CONDA_ROOT`、当前激活的 `MedRisk` 环境、常见 Anaconda/Miniconda 目录和 `conda env list` 发现同名环境。脚本不创建、不激活 `medrisk_model_service\.venv`，也不从随机 `PATH` Python 启动，并通过 `python -s` 与 `PYTHONNOUSERSITE=1` 降低用户级 Python 包对模型服务的干扰。

模型服务默认监听 `127.0.0.1:8090`，健康检查为 `GET /health`。依赖安装、服务 stdout/stderr 和启动进程记录均由脚本写入项目 `logs/` 目录。

## 管理员训练流程

1. 管理员上传结构化 CSV 或 ZIP 数据集，数据集必须包含 `label` 二分类目标列。
2. Spring Boot 将数据集保存到 `uploads/datasets/` 并记录校验结果。
3. 管理员创建训练任务后，FastAPI 读取共享数据集并训练 XGBoost 分类模型。
4. 训练产物写入 `models/training/job-{id}/`，包含 `model.joblib`、`history.json`、`metadata.json`。
5. 训练成功后后端写入新的 `model_versions` 记录，管理员可在管理后台启用该版本。
6. 启用后模型服务加载 `model.joblib`，对应病种预测优先走训练模型。

训练曲线来自 XGBoost `evals_result()`，前端展示 train/valid logloss 与 error。评估接口返回 Accuracy、Precision、Recall、F1、AUC、混淆矩阵和样本预测摘要。
