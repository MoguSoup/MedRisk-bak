# 部署说明

## Docker Compose

```powershell
copy .env.example .env
docker compose up --build
```

生产环境必须修改 `.env` 中的数据库密码、Neo4j 密码和 JWT 密钥。Compose 部署会启动 MySQL、Neo4j、模型服务、后端和前端；后端默认连接 Compose 内部的 `bolt://neo4j:7687`。Linux 云服务器 `med_tencent` 以 Docker Compose 为标准启动方式，不需要额外维护 Linux 原生启动脚本，也不推荐把远端 Neo4j Bolt `7687` 暴露到公网。

开源发布时不要直接把 `med_tencent:/home/med/MedRisk` 原样复制为发布包。远端运行目录可能包含 `.env`、上传文件、训练模型、日志、缓存、个人报告材料和历史乱码重复文档；公开发布应以本地 git 工作区为准，并按 [开源发布清单](开源发布清单.md) 逐项确认。

前端容器使用 Nginx 托管静态资源，并将 `/api/` 代理到 `http://backend:8080/api/`。因此生产访问只需要暴露前端端口；如需排查后端，可临时访问映射的 `8080` 健康检查。

Compose 会将本地 `uploads/` 和 `models/` 同时挂载给 Spring Boot 后端与 FastAPI 模型服务：

- 数据集：`uploads/datasets/`
- 知识文档、问答图片、疾病图片、病历图片：`uploads/` 下对应 bucket
- 训练产物：`models/training/`
- 已启用模型映射：`models/training/active-models.json`，由 `MEDRISK_ACTIVE_MODELS_FILE` 控制，用于模型服务重启后继续加载管理员启用的训练模型
- 公开训练/评估 CSV：`data/processed/`，容器内默认映射为 `MEDRISK_PUBLIC_DATASET_DIR=/shared/data/processed`
- 报告：Docker 命名卷 `medrisk_reports`
- Neo4j 图谱数据：Docker 命名卷 `medrisk_neo4j_data`

## 本地开发端口

| 服务 | 端口 |
|---|---:|
| 前端 Vite | 优先 5173；Windows 保留时自动回退到 5241 |
| Docker Compose 前端 Nginx | `.env` 中 `FRONTEND_PORT`，默认 5241；远端宿主机 Nginx 使用 80 代理到该端口 |
| Spring Boot 后端 | 8080 |
| FastAPI 模型服务 | 8090 |
| MySQL | 3306 |
| Neo4j Browser | 127.0.0.1:7474，仅服务器本机排障访问 |
| Neo4j Bolt | 127.0.0.1:7687，仅服务器本机排障访问；后端容器使用内部 `neo4j:7687` |

## 本地脚本运行

`scripts/start-all.ps1` 默认使用 Spring Boot `demo` profile 和 H2 文件库，适合本机快速试用；Docker Compose 使用 MySQL + Neo4j，适合验收部署。

本地脚本约定：

- 模型服务固定使用 conda `MedRisk` 环境。脚本默认兼容 `D:\IDEA\Anaconda\envs\MedRisk\python.exe`，也会通过 `MEDRISK_CONDA_ROOT`、当前激活的 `MedRisk` 环境、常见 Anaconda/Miniconda 目录和 `conda env list` 发现其他电脑上的同名环境。
- 不创建、不激活 `medrisk_model_service\.venv`，也不从随机 `PATH` Python 启动。
- 模型服务启动前会在 conda `MedRisk` 环境内安静执行 `python -s -m pip install -r requirements.txt`，依赖安装输出写入 `logs/model-service-install.log`，终端只保留启动状态和 ready 信息。
- 后端通过 `mvn clean spring-boot:run -Dspring-boot.run.profiles=demo` 启动，避免旧构建产物中的重复 Flyway 迁移文件影响启动。
- `demo` profile 只面向本地 H2 文件库，已关闭 Flyway checksum 校验以兼容历史演示库迁移记录；Linux `med_tencent` 的 Docker Compose/MySQL 路径不使用该宽松校验。
- 后端启动前会校验本地 Neo4j Bolt 地址，默认 `MEDRISK_NEO4J_URI=bolt://localhost:7687`。本地脚本只检查本机连接，不安装、不拉起、不停止 Neo4j；未就绪时会失败并提示启动本机 Neo4j 服务，或在 Docker Desktop 可用时运行下方本地容器命令。Windows 本地演示不会自动连接 `med_tencent`，也不要求开放远端 `7687`。
- `bolt://neo4j:7687` 只属于 Docker Compose 内部网络。若该值被复制到 Windows 本地 `.env` 或当前 PowerShell 环境，本地脚本会改用 `bolt://localhost:7687` 并提示；本地修复方式是启动本机 Neo4j，而不是连接远端 Neo4j。
- 前端优先绑定 `127.0.0.1:5173`；如果该端口被其他进程占用或被 Windows/Docker/Hyper-V 保留，脚本会自动选择 `5241` 等可绑定端口，并打印实际 URL。Docker Compose 前端端口由 `.env` 的 `FRONTEND_PORT` 控制，示例默认 `5241`；远端 `med_tencent` 的宿主机 Nginx 监听 80 并代理到该端口。
- 前端会检查 `vite`、`@vitejs/plugin-vue`、`jsdom`、`vue` 等依赖，不完整时自动安静执行 `npm install --silent`，安装输出写入 `logs/frontend-install.log`。
- 后台进程的 stdin 被重定向到 `logs/empty.stdin`，脚本结束后当前 PowerShell 终端应可继续输入命令。
- 启动脚本会等待模型服务 `/health`、后端 `/api/health` 和前端页面均可访问后才输出 `MedRisk AI services are ready`；如果某项超时，会打印相关日志尾部并返回失败。
- 重复执行 `scripts/start-all.ps1` 时会复用已登记或已运行的前端 fallback 端口，避免 5173 不可绑定时在 5241、5484、5500 等端口重复拉起 Vite。

### 反复启动故障：登录 500 / 后端 8080 未启动

本地演示库 `data/medrisk-demo` 可能已经记录旧版 Flyway 迁移 checksum；当迁移文件内容变化后，若 `demo` profile 恢复默认校验，Spring Boot 会在启动阶段报 checksum validation failed，后端 8080 不监听，前端登录表现为 500。防复发规则：

- `medrisk_backend/src/main/resources/application-demo.yml` 必须保留 `baseline-on-migrate: true` 和 `validate-on-migrate: false`。
- 该配置只允许用于本地 H2 demo；`application.yml`、Docker Compose 和 Linux `med_tencent` 的 MySQL 部署不得关闭 Flyway 校验。
- 本地脚本必须继续使用 `Resolve-MedRiskPython`、conda `MedRisk` 和 `mvn clean spring-boot:run -Dspring-boot.run.profiles=demo`，不得回退到 `.venv`、随机 `PATH` Python、旧 `Resolve-CondaPython` 或非 `clean` 启动。
- 若再次出现登录 500，优先检查 `logs/backend.out.log` 是否有 Flyway checksum 报错，再检查 `http://localhost:8080/api/health` 是否返回 `UP`。

常用命令：

```powershell
.\scripts\start-all.ps1
.\scripts\stop-all.ps1
.\scripts\ci.ps1
```

`scripts/stop-all.ps1` 会清理记录在 `logs/medrisk-processes.json` 的进程、5173/8080/8090 端口进程，以及本项目目录下残留的 Vite/esbuild、Maven/Spring Boot、Uvicorn 相关进程。
它不会停止本地外置 Neo4j。若使用单独的 Docker Neo4j 容器，请按你的容器名称手动停止，例如 `docker stop medrisk-neo4j`。

可选的本地 Neo4j 快速启动命令：

```powershell
docker run --name medrisk-neo4j -p 7474:7474 -p 7687:7687 -e NEO4J_AUTH=neo4j/medrisk-neo4j -d neo4j:5.26-community
```

智能问答 LLM 通过 OpenAI-compatible 接口配置：`MEDRISK_LLM_BASE_URL`、`MEDRISK_LLM_API_KEY`、`MEDRISK_LLM_MODEL`、`MEDRISK_VISION_MODEL`、`MEDRISK_IMAGE_MODEL`。未配置 API Key 时，后端会基于图谱/疾病/病历检索结果生成演示兜底答案。使用阿里云百炼时，同一个 API Key 可覆盖文本问答、图片输入和图片输出，系统按场景自动切换到对应模型。

阿里云百炼 workspace 专属域名可直接作为兼容地址，例如 `MEDRISK_LLM_BASE_URL=https://your-workspace.cn-beijing.maas.aliyuncs.com/compatible-mode/v1`、`MEDRISK_LLM_MODEL=qwen-plus`。真实 API Key 只能写入本地或服务器 `.env`，不要写入 `.env.example`、README、文档、源码、远端命令日志或前端构建产物。

问答接口会先执行平台范围判断：医疗健康、疾病风险预测、医学知识库、病历/文档、知识图谱、模型训练评估和平台使用相关问题才会进入检索与外部模型；明显无关问题直接返回范围提示，`usedModel=policy-guard`，不会调用外部 LLM。

如使用 cockpit-tools v0.26.5 一类 provider gateway/反向代理网关，把后端指向该网关的 OpenAI-compatible `/v1` 地址即可，例如 `MEDRISK_LLM_BASE_URL=http://127.0.0.1:8787/v1`、`MEDRISK_LLM_API_KEY=<gateway-token>`、`MEDRISK_LLM_MODEL=<gateway-model>`。只允许写入网关 URL、token 和模型名；不要把 ChatGPT 网页登录邮箱、密码、cookie 或浏览器会话写入 `.env`、文档、源码或远端命令日志。

注册邮箱验证码和忘记密码邮件使用 SMTP 配置：`MEDRISK_MAIL_HOST`、`MEDRISK_MAIL_PORT`、`MEDRISK_MAIL_USERNAME`、`MEDRISK_MAIL_PASSWORD`、`MEDRISK_MAIL_FROM`、`MEDRISK_MAIL_SSL`、`MEDRISK_MAIL_STARTTLS`、`MEDRISK_MAIL_AUTH`。163 邮箱固定推荐 `smtp.163.com:465 + SSL`，即 `MEDRISK_MAIL_SSL=true`、`MEDRISK_MAIL_STARTTLS=false`；`MEDRISK_MAIL_PASSWORD` 必须是第三方客户端授权码，不能写网页登录密码。未配置 SMTP 时应用仍可启动，但发送验证码接口会返回明确业务错误。

模型服务默认依赖保证 scikit-learn 基线、XGBoost、LightGBM、CatBoost 和 TabPFN 可用，包括 Logistic Regression、RandomForest、ExtraTrees、HistGradientBoosting、XGBoost、LightGBM、CatBoost、TabPFN。Docker 镜像会先从 PyTorch CPU wheel 源安装 `torch==2.6.0`，再安装 `requirements.txt`，避免默认 PyPI 拉取 CUDA/GPU 大依赖导致云服务器磁盘不足。TabPFN 默认使用 `MEDRISK_TABPFN_VERSION=v2`，如果显式切换到 v3，需要先接受 Prior Labs/Hugging Face 条款并配置 `HF_TOKEN`、`HUGGINGFACE_HUB_TOKEN` 或 `TABPFN_TOKEN`。TabICL 和 FT-Transformer 仍属于高级可选能力，可在 conda `MedRisk` 环境或镜像构建流程中按需安装 `medrisk_model_service/requirements-advanced.txt`；未安装时 `/models/capabilities` 会返回不可用原因，系统仍可正常启动和训练默认模型。

预测模型不是仅前端展示的占位项。默认基线以已部署的 XGBoost 风险模型形式提供预测接口，并在模型版本页展示公开评估来源：糖尿病、心脏病、慢性肾病和中风参考 CDC BRFSS 2024 派生数据，肝病参考 CDC NHANES 2017-March 2020 派生数据。小样本 UCI 不再默认导入或展示。管理员上传 CSV 或导入公开 CSV 后完成训练、评估并点击启用，模型服务会把该病种的活动模型写入 `MEDRISK_ACTIVE_MODELS_FILE`；只要对应的 `model.joblib` 仍在 `models/training/` 挂载目录中，容器重启后预测接口会继续使用该启用版本。

大型公开数据集不提交仓库。需要复现公开训练/评估时，在本地或远端运行 `scripts/prepare-large-datasets.ps1` 生成 `data/processed/manifest.json`、训练 CSV 和评估 CSV，然后在管理员“数据集管理”点击“导入公开训练数据集”。该动作按 `sourceRecordId` 幂等注册，不会重复膨胀数据集列表。MIMIC-IV 只支持读取已授权的本地目录：设置 `MEDRISK_MIMIC_IV_DIR` 后，脚本会从 `hosp/admissions`、`patients`、`diagnoses_icd` 生成五病种 CSV；未设置时会跳过，不影响 CDC 数据集。

生产环境若经过 Nginx、SLB 或云负载均衡转发，请保留 `X-Forwarded-For` 或 `X-Real-IP`，后端登录/退出审计会优先记录这些代理头中的第一个有效 IP，再回退请求远端地址。

`V7` 数据库迁移会创建邮箱验证码表，并按需求自动清空旧 `audit_logs`。部署前如需保留旧审计日志，请先从数据库自行导出备份；迁移后所有新 HTTP 操作都会自动补齐请求 IP。

## 演示数据与图谱导入

后端启动时会幂等补齐基础演示包，不会重复插入已经存在的 `sourceRecordId`。管理员可在前端“数据源管理”中手动执行：

- “补齐演示数据”：写入关系库中的疾病、文档和合成病历；训练/评估数据集请通过公开数据脚本、MIMIC-IV 本地预处理或管理员上传导入。
- “重建演示包”：补齐关系库数据并尝试向 Neo4j 写入结构化三元组。

演示包以公开来源元数据和本地合成内容组成，覆盖 MedlinePlus、Disease Ontology、HPO、openFDA 链接型知识，以及 Synthea 风格合成病历。病历均标注“合成数据，非真实患者”，不包含真实姓名、证件号、电话、住址等身份信息。Neo4j 清理和重建只处理 `medriskSource=MedRisk` 的节点与关系，避免误删外部图谱数据。

知识文档图谱构建使用离线规则抽取疾病、症状、风险因素、检查、治疗、药物、文本单元、关键词和观点节点。每篇非空文档会写入文档级唯一 `graphKey`，避免不同文档同名实体互相覆盖；单文档同步、增量构建和全量重建都会为单篇文档生成至少 50 个文档相关节点。增量构建会同时处理 `未构建` 与 `构建失败` 文档，批量任务中单篇失败会被标记为失败但不会中断其他文档。

文档、疾病、病历和图谱数据使用 `visibility` 做分级：`PUBLIC` 面向患者/医生/管理员，`DOCTOR_ONLY` 面向医生/管理员，`ADMIN_ONLY` 仅管理员，`DRAFT` 用于医生草稿和待发布内容。

换电脑部署本地开发环境时，需要先安装 Java 17、Maven、Node.js 20+ 和 Anaconda/Miniconda，并创建 conda 环境 `MedRisk`。如果 conda 根目录不在常见位置，可在当前 PowerShell 设置：

```powershell
$env:MEDRISK_CONDA_ROOT = "D:\Tools\miniconda3"
.\scripts\check-env.ps1
```

## 注意事项

- 系统只用于教学演示，不接入真实医院数据。
- 报告与模型输出必须保留免责声明。
- 示例数据应来自公开数据源元数据、许可证清晰的数据集或合成数据。
- 上传的数据集必须脱敏，且至少包含 `label` 目标列。
- 训练模型启用前应查看评估指标、混淆矩阵和模型反馈记录。
- 个人工作日志、实习报告、AI 工具记录、远端 `.env`、训练产物和受控数据集不得随开源版本发布。
